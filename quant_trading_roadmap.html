開始之前：待確認事項

⚠ 請在啟動前確認以下 4 項
01
Python 熟練程度
量化工作幾乎全用 Python（pandas、numpy、matplotlib）。如果你主要背景是 Java 或 Go，需要額外花 1-2 週熟悉 pandas 的向量化數據操作——這不是語法問題，是思維方式的切換。如果已經熟練，這一條跳過。
02
每週可投入時間
本路線圖所有時間估算基於 5-10 小時 / 週。你目前在讀碩士 + 接 Upwork + 家庭，如果每週只有 3-4 小時，所有時間線乘以 1.5 倍，不要逼自己。
03
Binance / Alpaca 帳戶狀態
Phase 0 需要建立 Binance Testnet 和 Alpaca Paper Trading 帳戶並拿到 API Key。Binance Testnet 不需要 KYC，5 分鐘搞定。Alpaca Paper 免費帳戶同樣簡單。如果已有正式帳戶，Testnet Key 單獨申請即可。
04
Paper Trading 模擬資金規模
建議設定為你計劃實際投入的金額（例如 $10,000 USD）。理由不是準確性，而是讓你從一開始就用真實規模計算倉位，養成習慣。

整體邏輯與核心原則

你在 Paperclip 遇到的管理混亂，根本原因是沒有外部記憶：不知道每個 agent 在幹什麼，不知道自己試過什麼，不知道為什麼做了這個決定。

解決方案不是換一個更好的框架，而是建立一套讓所有決策可見、可回溯、可測量的工作方式。Git + 代碼 + 研究日誌，就是這套方式。

Monorepo 決策

✓ 結論：合理，從學習到實盤全用一個倉庫。有明確的充分理由再拆，不要提前拆。

為什麼合理

  • 學習代碼 → 研究代碼 → 實盤代碼是自然演進的關係，分拆只會製造同步問題
  • Git history 就是你的完整研究記錄，拆庫會斷掉這個連續性
  • 策略指標代碼在學習階段寫好，後續直接 import，不需要跨庫引用
  • 你是獨立開發者，monorepo 的協作管理成本接近零

什麼時候才需要拆

  • 進入實盤後需要在 VPS 部署，只部署 05_live/ 目錄即可，不需要拆庫
  • 真正需要拆的情況：引入協作者，或有 CI/CD 安全合規要求
  • 保守估計：至少一年後才可能需要

Phase 1：業務學習

⏱ 3-4 週 · 10 個概念腳本 · 每週 2-3 個

這個階段的核心原則：每個概念，用代碼驗證理解，不是用文字記憶。 Claude Code 生成腳本，你主動改，問自己能不能解釋每一行。

Claude Code 學習工作流（每個概念重複這個 5 步流程）

1
給 Claude Code 的 Prompt 模板

「生成一個最小可運行的 Python 腳本，用真實 BTC/USDT 日線數據演示 [概念名稱]。要求：每一行都加業務含義的注釋（不是技術說明），輸出要有可視化圖表，代碼不超過 60 行。」

2
跑起來，看輸出是否在直覺上合理

例：EMA 線應該比 SMA 對最近價格更敏感，圖上能看出來嗎？ATR 在大波動期間應該更高，是嗎？

3
主動改動（至少選一個）

① 改一個參數，先預測結果再跑驗證（例：EMA 週期從 20 改到 5，線會怎樣？）
② 注釋掉一行，看報錯信息，搞清楚這行在幹什麼
③ 換一個資產（ETH 或 SPY），規律是否一樣成立？

4
自我測試：能回答這三個問題嗎？

① 這個指標在業務上代表什麼意思？
② 它的局限性是什麼？在什麼市場環境下會失效？
③ 如果用它做交易信號，最大的風險是什麼？

5
Commit（帶有意義的 message）

格式：learn: [概念] - [一行關鍵洞察]
例：learn: ATR - period=14 smooths noise well, period=7 overfits recent vol

Phase 2：策略研究

⏱ 4-8 週 · 目標 20+ 次有記錄的實驗

這個階段不寫新的基礎設施，只做一件事：系統性地提出假設、驗證假設、記錄結論。 每次實驗都是科學實驗，不是亂試一通。

首先：搭建回測引擎（Phase 2 第一週）

讓 Claude Code 幫你實現 03_research/backtest/engine.py，但你需要理解並能修改以下每一個組件：

組件具體要求為什麼不能省略
手續費模擬 Binance Taker 0.1%，Alpaca 免佣但有 bid-ask spread（加 0.03% 估算） 忽略手續費的回測結果通常虛高 20-40%，高頻策略更明顯
滑點模擬 每筆交易加 0.05% 保守估算 實盤執行價格永遠比回測的收盤價差，日線策略影響相對小但不能忽視
固定風險倉位 每筆風險 = 賬戶 × 1-2%，止損距離 = 2×ATR，計算買入數量 這是控制回撤的核心。倉位相同的策略，回撤可以差 3 倍
樣本劃分 前 70% 樣本內（調參），後 30% 樣本外（驗證）。樣本外數字才算數 防止在測試集上優化參數，這是過擬合最常見的成因
標準化輸出 JSON 格式績效報告（方便填入 STRATEGY_LOG） + 淨值曲線圖 統一格式讓不同實驗可以直接對比

實驗工作流（每次實驗重複）

1
在 STRATEGY_LOG.md 先寫假設，再動代碼

「加入 ADX>25 過濾條件能減少震盪市假信號」是假設。「我想試試 ADX」不是假設。先寫再跑，強迫自己想清楚在驗證什麼。

2
建立 experiments/exp_XXX/ 文件夾

複製上一個實驗的 config.py 修改參數，不是從零生成。讓 Claude Code 幫你改，但你要知道改了什麼。

3
跑回測，保存 results.json

同時跑樣本內和樣本外。只看樣本內結果等於欺騙自己。

4
寫 notes.md，更新 STRATEGY_LOG.md

必須寫結論，哪怕是「假設不成立，原因是 X」。失敗的實驗和成功的一樣有價值——它告訴你哪條路走不通。

5
Commit

格式：research: exp_XXX [策略名] - OOS Sharpe=X.X MaxDD=-XX%
把樣本外的關鍵數字放進 commit message，Git log 就是你的實驗摘要。

Phase 3：Paper Trading

⏱ 6-8 週 · 系統完全自動化運行

Paper Trading 的目標不是賺紙面盈利，而是發現所有回測看不見的工程問題：API 斷線、數據延遲、訂單被拒、交易所維護窗口。這裡出問題的代價比實盤低 100 倍。

Agent 架構（04_paper_trading/agents/）

Agent職責輸入 → 輸出
data_agent 實時數據拉取與緩存 WebSocket 訂閱 → 標準化 OHLCV DataFrame
signal_agent 在最新數據上運行策略 DataFrame → SignalEvent（方向、強度、標的）
risk_agent 風控審核，硬性規則，不可繞過 SignalEvent → OrderEvent 或 RejectionEvent
execution_agent 執行訂單，記錄結果 OrderEvent → FillEvent（或 FailEvent + 報警）

風控 Agent 硬性規則

  • 單筆最大虧損：賬戶的 1.5%。超過此規模的訂單直接拒絕，不需要人工干預
  • 每日最大虧損熔斷：當日累計虧損 > 4% → 停止當日所有交易，發 Telegram 報警
  • 最大同時持倉數：加密貨幣 3 個，美股 5 個
  • 相關性限制：新倉與現有持倉相關係數 > 0.8 時拒絕開新倉（避免名義上分散實際上集中）
  • 數據異常保護：若數據超過 15 分鐘未更新 → 暫停信號生成，發報警，等確認後才繼續

核心績效指標速查

指標定義趨勢跟蹤的一般標準
Sharpe Ratio 年化超額收益 / 年化波動率 > 1.0 才考慮進 Phase 3，> 2.0 是很好的策略
Max Drawdown 從峰值到谷底的最大連續虧損 趨勢跟蹤通常 15-25%，超過 30% 需要重新看風控
Win Rate 盈利交易數 / 總交易數 35-50% 正常。低勝率 + 高盈虧比 = 可以盈利
Profit Factor 總盈利 / 總虧損 > 1.5 才有意義，> 2.0 是健康的策略
OOS Decay 樣本外 Sharpe / 樣本內 Sharpe > 0.6 說明過擬合程度可接受，< 0.5 需要重新設計

常見 Bug 與對應檢查

現象最可能的原因怎麼查
回測年化 > 80% 前視偏差 確認信號計算時用的是前一天收盤價，不是當天
樣本外 Sharpe 比樣本內低一半以上 過擬合 減少可調參數，只保留有業務含義的
Paper 績效遠差於回測 手續費/滑點低估 對比每筆交易的成本假設
某天突然大虧 風控沒有生效 檢查 risk_agent 的日誌，確認觸發條件是否正確
